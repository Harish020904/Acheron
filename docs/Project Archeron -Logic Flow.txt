# Project Acheron — Level 4 Technical Specification

## Million-Agent Urban Simulation Kernel

### C++20/23 Systems Architecture

---

# 1. The Hierarchical ECS Kernel (The Backbone)

The ECS is not an “entity system.”
It is a cache-optimized memory database.

At 1,000,000+ agents:

* inheritance fails
* virtual dispatch fails
* pointer chasing fails
* heap allocation fails

The ECS must optimize:

* contiguous iteration
* SIMD traversal
* prefetch predictability
* deterministic memory ownership

---

## 1.1 Data Layout

The architecture uses:

* Archetype Chunks
* Struct-of-Arrays (SoA)
* 16KB fixed chunks
* Manual alignment
* Stable type metadata

Do NOT use:

* `std::vector<BaseComponent*>`
* RTTI
* polymorphic hierarchies

---

## Core Memory Model

Each archetype owns:

* a fixed chunk list
* contiguous component columns
* entity metadata

Each chunk:

* stores entities sharing identical component signatures

Example archetype:

```cpp
Position + Velocity + TrafficAgent + EconomicState
```

---

## Chunk Layout

```cpp
constexpr size_t CHUNK_SIZE = 16 * 1024;

struct alignas(64) ArchetypeChunk
{
    std::byte* componentBuffers[MAX_COMPONENTS];

    uint32_t entityCount;
    uint32_t capacity;

    EntityID entities[CHUNK_ENTITY_CAP];

    ArchetypeChunk* next;
};
```

---

## SoA Component Layout

BAD:

```cpp
struct Agent
{
    Position p;
    Velocity v;
    Traffic t;
};
```

This destroys SIMD traversal.

---

CORRECT:

```cpp
struct PositionArray
{
    alignas(64) float x[CHUNK_ENTITY_CAP];
    alignas(64) float y[CHUNK_ENTITY_CAP];
};

struct VelocityArray
{
    alignas(64) float vx[CHUNK_ENTITY_CAP];
    alignas(64) float vy[CHUNK_ENTITY_CAP];
};
```

Now AVX2 can process:

* 8 floats simultaneously
* sequential memory
* no branch divergence

---

## Manual Allocation with std::byte

```cpp
struct ComponentColumn
{
    std::byte* buffer;
    size_t stride;
    size_t alignment;
};
```

Allocation:

```cpp
buffer = static_cast<std::byte*>(
    std::aligned_alloc(alignment, stride * capacity)
);
```

---

## Hardware Consequence

Contiguous SoA traversal:

```cpp
for (size_t i = 0; i < count; i += 8)
```

allows:

* hardware prefetchers
* SIMD vector loads
* fewer TLB misses
* reduced L1 eviction

AoS layouts force:

* scattered memory fetches
* partial cache line usage
* branch-heavy traversal

# 2. Massive-Scale Concurrency (The Heartbeat)

---

# 2.1 Fiber-Based Job System

The scheduler is not thread-oriented.

It is:

* task-oriented
* dependency-oriented
* work-stealing

Threads are workers only.

---

## Core Architecture

Worker count:

```cpp
const uint32_t workers =
    std::thread::hardware_concurrency();
```

Each worker owns:

* local deque
* fiber pool
* scratch allocator

---

## Job Structure

```cpp
struct Job
{
    void (*entry)(void*);
    void* data;

    std::atomic<uint32_t> unfinishedJobs;
};
```

---

## Fiber Model

Each job:

* suspends without blocking OS thread
* resumes later
* avoids kernel scheduler overhead

Critical for:

* pathfinding
* streaming
* async dependency waits

---

## Work Stealing

Workers:

* pop locally from bottom
* steal remotely from top

Deque:

```cpp
alignas(64) std::atomic<uint64_t> top;
alignas(64) std::atomic<uint64_t> bottom;
```

This reduces:

* lock contention
* false sharing
* global queue bottlenecks

---

## Mapping 1,000,000 Agents

DO NOT:
schedule per-agent jobs.

Instead:

```cpp
1 job = 4096 agents
```

Example:

```cpp
for (size_t i = 0; i < agents; i += 4096)
{
    Schedule(UpdateAgents, range);
}
```

---

## Hardware Consequence

Chunk scheduling:

* amortizes scheduler overhead
* improves instruction cache reuse
* improves branch predictor stability

Per-agent jobs would destroy:

* context locality
* queue contention
* scheduler throughput

---

# 4.1 Continuum Traffic Model

Vehicles represented as density fields.

Not:
individual rigid bodies.

---

## Road Segment

```cpp
struct alignas(64) RoadChunk
{
    float density[LANES];
    float velocity[LANES];
    float flowRate[LANES];
};
```

---

## Fundamental Equation

Traffic flow:

q = \rho v

Where:

* q = flow
* ρ = density
* v = average velocity

---

## Velocity Function

v = v_{max}(1 - \rho/\rho_{max})

---

## Update Function

```cpp
void UpdateRoad(RoadChunk& r, float dt)
{
    for (size_t i = 0; i < LANES; ++i)
    {
        r.velocity[i] =
            vmax * (1.0f - r.density[i] / maxDensity);

        r.flowRate[i] =
            r.density[i] * r.velocity[i];

        r.density[i] +=
            dt * (incoming[i] - outgoing[i]);
    }
}
```

---

## Hardware Consequence

Dense float arrays:

* vectorize naturally
* fit AVX lanes
* predictable memory access

Rigid-body traffic:

* branch-heavy
* collision-heavy
* impossible at city scale

# 5.2 Memory Safety & Allocators

The default heap allocator will fragment catastrophically.

You need:

* frame allocators
* arenas
* monotonic allocators

---

## Monotonic Allocator

```cpp
struct LinearAllocator
{
    std::byte* memory;
    size_t offset;
    size_t capacity;
};
```

Allocate:

```cpp
void* Allocate(size_t size, size_t align)
{
    size_t aligned =
        (offset + align - 1) & ~(align - 1);

    void* ptr = memory + aligned;

    offset = aligned + size;

    return ptr;
}
```

Reset once per frame.

No frees.

---


Benefits:

* bounds-safe views
* no ownership ambiguity
